# AGENTS.md

本项目的协作规范与改造指南，服务于后续四步改造目标：
- 测试设计缺口提示
- 联动影响分析
- 知识关系增强
- 字段一致性治理

## 0. 适用范围
本文件适用于本仓库的所有改动，包括但不限于：
- Streamlit 页面层（`apps/streamlit_views.py`）
- 平台编排层（`apps/testcase_platform.py`）
- 检索与生成（`src/rag/generate_testcase.py` / `workflow_graph.py`）
- 知识库写入与索引（`src/rag/kb_upsert.py`）

---

## 1. 工作方式原则（必须遵守）

### 1.1 先计划后实现
- 任何功能改动，先提交**修改计划**（范围、影响、验证）。
- 计划必须明确：**涉及文件、具体修改点、验证方法、风险说明**。
- 未完成计划确认前不得直接动代码。

### 1.2 最小必要改动
- 优先“低风险、局部改动、可回滚”。
- 禁止无关重构、架构推翻、风格大改。
- 新能力应尽量复用现有数据结构与流程。

### 1.3 页面层禁止堆业务逻辑
- UI 仅负责展示、交互、轻量格式化。
- 新的业务逻辑必须落在平台层或 RAG / workflow 层。
- 页面层新增逻辑必须是“可视化/格式化/交互收集”类。

### 1.4 过滤 / metadata / ext_* 映射 必须补测试
- 凡涉及 `metadata`、`ext_*` 映射或检索过滤条件，必须同步更新/补充测试。
- 至少覆盖：`approved_only`、`release`、`modules` 的过滤逻辑。
- 推荐落地在 `tests/` 中的单元测试（例如：`tests/test_workflow_graph.py` 或新增轻量测试）。

### 1.5 Done 定义（强制输出）
任何改动完成必须输出：
- **修改说明**：改了什么、为什么改
- **验证方法**：如何验证
- **风险说明**：可能影响与边界

## 2. 四步改造目标与约束

### 2.1 测试设计缺口提示
目标：在生成或评审阶段提示缺失的覆盖点。
- 允许改动：`workflow_graph.py` 的评审产出结构、`apps/streamlit_views.py` 展示区轻量提示。
- 禁止改动：页面层直接做“业务规则判断”。
- 建议方案：在评审结果中新增缺口列表（不改变主结构，使用已有 `review_result` 或新增字段）。
- 测试要求：评审输出结构变更需补测试，确保 JSON 解析与显示不崩。

### 2.2 联动影响分析
目标：提示跨模块联动影响与上下游风险。
- 允许改动：检索上下文组织方式、生成 prompt 合成逻辑。
- 禁止改动：引入复杂外部依赖或分布式调用。
- 建议方案：在 `generate_testcase` 中增强上下文组织（但保持最小改动）。
- 测试要求：检索过滤逻辑变更必须补测试。

### 2.3 知识关系增强
目标：让生成结果更多关联历史用例/需求/接口文档。
- 允许改动：检索 query 组合策略、候选筛选策略（保持小改动）。
- 禁止改动：更换向量库或大范围结构调整。
- 建议方案：为不同 source_type 适度加权或增加轻量关系提示。
- 测试要求：新增检索策略需覆盖至少一个单测样例。

### 2.4 字段一致性治理
目标：消除 metadata/状态字段不一致导致的过滤偏差。
- 允许改动：`kb_upsert.py` 中 ext_* 映射规则；`generate_testcase.py` 的过滤函数。
- 禁止改动：改变已有数据结构存储（非必要重写）。
- 必须补测试：新增字段映射或过滤规则必须有测试覆盖。

---

## 3. 约束清单（必须遵守）

- 不改变日志写入结构（除非明确要求），展示层可优化但不得丢失原始信息。
- 不删除原始 JSON 输出能力，只能折叠或层级优化。
- 不引入不可控外部依赖（如需，必须提前评审）。
- 代码改动必须可回滚且不影响现有主流程。

---

## 4. 推荐的提交流程（简化版）

1. **提出修改计划**  
   - 涉及文件  
   - 具体改动点  
   - 验证方法  
   - 风险说明  

2. **确认最小改动范围**  
   - 是否能复用现有字段结构  
   - 是否能不改主流程  

3. **实现与测试**  
   - 涉及 ext_* / metadata / 过滤逻辑 → 必补测试  

4. **完成反馈（Done 定义）**  
   - 修改说明  
   - 验证方法  
   - 风险说明
