# 模块本体定义文档

## 1. 目的
用于统一系统中的一级模块、二级能力、模块别名和模块间关系，作为以下能力的规则来源：

- 知识入库 metadata 规范化
- 当前需求涉及模块识别
- 潜在联动模块分析
- 测试样本标注
- 页面展示统一命名

本文件的目标不是描述系统架构，而是为“测试设计辅助平台”提供一套稳定、可解释、可维护的模块语义规则。

---

## 2. 模块划分原则

### 2.1 一级模块以产品导航为准
当前系统一级模块按产品左侧导航定义，不再额外抽象“业务域”。

当前一级模块为：

- 生产
- 项目
- 订单
- 营销
- 店铺
- 会员
- 核销
- 商品
- 报表
- 财务
- 多语言

说明：
- “首页”不作为业务模块参与归因与联动分析。
- 一级模块是系统识别、联动分析、样本标注的最高层标准模块。
- 若后续左侧导航发生调整，应同步更新本文件与配置文件。

### 2.2 二级能力从属于一级模块
每个一级模块下可继续定义二级能力，用于更细粒度理解需求，但模块归因默认先落到一级模块。

例如：
- 订单：手动下单、售后单、订单管理、订单状态流转、履约
- 会员：会员等级、积分、权益、会员标签、权益卡
- 财务：结算、对账、交易流水、发票
- 核销：核销规则、核销记录、核验监控

说明：
- 二级能力用于增强理解，不直接替代一级模块。
- 默认先识别一级模块，必要时再补充二级能力信息。

### 2.3 不允许随意合并真实独立模块
如果两个能力在实际产品中属于不同一级模块，则不能为了评测方便临时合并。

例如：
- 会员权益属于“会员”
- 商品库存若存在，应归入“商品”或其明确子能力
- 不允许把“商品库存”和“会员权益”合并成同一个模块

说明：
- “有关系”不等于“是同一个模块”
- “会互相影响”也不等于“可以合并建模”

### 2.4 模块关系单独定义
模块之间存在互相影响是正常现象，但“有关系”不等于“是同一个模块”。

例如：
- 订单 可能影响 财务
- 营销 可能影响 订单
- 会员 可能影响 订单或营销
- 商品 可能影响 订单
- 订单 可能影响 核销

这些应通过“关系边”表达，而不是通过模块合并表达。

## 3. 一级模块定义

### 3.1 生产
- 标准名：生产
- 说明：生产项目管理、生产出票、出票单
- 常见识别词：生产、出票、出票单、生产出票

补充边界：
- 生产偏“执行与产出”
- 项目偏“定义与配置”
- 两者可相关，但不是同一模块

### 3.2 项目
- 标准名：项目
- 说明：销售项目管理、外部项目管理、供应商管理、预约管理、跨场次套票、报名投票、分销商管理
- 常见识别词：项目、销售项目、外部项目、供应商、预约、跨场次套票、报名投票、分销商

补充边界：
- 项目偏“业务对象与配置管理”
- 不等于订单，不等于生产执行

### 3.3 订单
- 标准名：订单
- 说明：订单管理、订单售后、订单履约、状态流转、手动下单
- 常见识别词：订单、下单、手动下单、售后单、履约、订单状态

### 3.4 营销
- 标准名：营销
- 说明：营销活动、优惠券、加购促销、特权购
- 常见别名：促销、活动、优惠、营销中心、优惠券、特权购

### 3.5 店铺
- 标准名：店铺
- 说明：店铺配置、渠道管理、店铺项目管理、店铺装修、店铺计划、巡演活动、消息推送
- 常见识别词：店铺、门店、渠道、店铺装修、店铺计划、巡演活动

说明：
- 若“消息推送”仅为店铺内配置项，则仍归属店铺模块
- 若未来存在独立消息系统，应单独定义模块，不继续放在店铺说明中

### 3.6 会员
- 标准名：会员
- 说明：会员信息、会员等级、积分、会员权益、标签、权益卡、签到有礼
- 常见识别词：会员、会员等级、积分、权益、标签、权益卡、签到有礼

说明：
- “权益”默认归属会员模块，除非未来另有明确模块定义
- 不允许把会员权益和商品库存混成一个模块

### 3.7 核销
- 标准名：核销
- 说明：核销项目管理、核销记录、核销规则、核销监控配置、核销优惠券、设备管理、内容管理
- 常见识别词：核销、验券、验码、核销记录、核销规则、核验监控、核销优惠券、设备管理

说明：
- 若“内容管理”仅指核销场景下的配置内容，则保留在核销模块
- 若未来“内容管理”成为通用能力，应拆出独立模块，不继续保留在核销说明中

### 3.8 商品
- 标准名：商品
- 说明：商品信息、SKU、商品属性、商品配置、商品标签、组合商品
- 常见识别词：商品、SKU、商品属性、组合商品、商品标签

说明：
- 商品库存若存在，默认作为商品的子能力处理，除非未来拆出独立库存模块

### 3.9 报表
- 标准名：报表
- 说明：统计报表、经营分析、明细查询、数据看板
- 常见识别词：报表、统计、分析报表、明细查询、数据看板

说明：
- 报表更多是聚合与呈现层
- 报表通常依赖多个业务模块的数据变化

### 3.10 财务
- 标准名：财务
- 说明：结算、对账、交易流水、发票
- 常见识别词：财务、结算、对账、交易流水、发票

### 3.11 多语言
- 标准名：多语言
- 说明：语言包、翻译配置、多语言展示与切换
- 常见别名：国际化、i18n、翻译、多语种

说明：
- 多语言用于描述展示与文案切换能力
- 不直接等同于业务模块，但作为一级模块参与识别和联动关系定义

## 4. 模块关系定义原则

### 4.1 关系类型
模块间关系统一分为三类：

- upstream：当前模块依赖的上游模块
- downstream：当前模块变化后更可能影响的下游模块
- related：存在强关联，但不直接表示明确影响方向

### 4.2 关系使用规则
- current module 识别优先基于需求直证
- linked module 分析才消费模块关系
- 不能因为存在关系边，就把 linked 模块抬升为 current module
- 关系边用于“补测联动提示”，不用于替代当前模块识别
- 若关系方向尚不明确，优先使用 `related`，不要写成混合类型

### 4.3 典型关系示例
以下仅为初始化示例，实际以业务确认后为准：

- 项目 -> 订单 （downstream）
- 商品 -> 订单 （downstream）
- 营销 -> 订单 （downstream）
- 会员 -> 订单 （related）
- 店铺 -> 订单 （downstream）
- 订单 -> 核销 （downstream）
- 订单 -> 财务 （downstream）
- 生产 -> 项目 （related）
- 核销 -> 订单 （related）
- 多语言 -> 项目 （related）
- 多语言 -> 商品 （related）
- 多语言 -> 店铺 （related）
- 多语言 -> 营销 （related）
- 报表 <- 项目 / 订单 / 营销 / 会员 / 商品 / 核销 / 财务 / 店铺 （upstream 聚合来源）

说明：
- 以上关系仅表示“测试时可能需要关注的联动方向”
- 不表示系统架构上的绝对调用关系
- 不表示模块本体归属关系

## 5. 测试使用规则

### 5.1 当前需求涉及模块
定义：
- 需求文本本身直接在讲的一级模块
- 最多输出 2 个
- 没有需求直证不得进入 `current_involved_modules`

使用原则：
- 当前模块优先由需求文本直接证据决定
- metadata、关系边、上下文分布只能增强，不得从 0 拉起 current module

### 5.2 潜在联动模块
定义：
- 不属于当前需求直接模块，但可能因关系边而受影响的模块
- 必须满足：触发模块 + 关系边 + 证据锚点
- 最多输出 2 个

使用原则：
- linked module 只能从 current module 出发推断
- linked 不得回指 current module
- 证据不足时宁可不输出

### 5.3 二级能力处理
- 识别到二级能力时，默认映射到所属一级模块
- 如后续需要更细粒度测试设计，可在一级模块下增加 `sub_capability` 字段，不改变一级模块口径

### 5.4 二级能力冲突处理规则
当同一个二级能力可能归属于多个一级模块时，按以下顺序处理：

1. 优先依据需求文本上下文判定所属一级模块  
   - 若需求文本中已明确说明业务对象或操作场景，则以该上下文为准。

2. 若需求文本不足以判定，则优先使用显式 metadata  
   - 如 asset metadata、ext_module、人工标注模块等。

3. 若仍存在冲突，不自动提升为 `current_involved_modules`  
   - 仅作为辅助证据参与打分，不直接进入当前需求涉及模块。

4. 若该二级能力与某一级模块存在稳定从属关系，应在模块本体配置中显式登记  
   - 例如：
     - 权益卡 -> 会员
     - 核验监控 -> 核销
     - 手动下单 -> 订单

5. 不允许因评测方便，临时修改二级能力归属  
   - 若归属发生变化，必须同步更新：
     - 模块本体配置
     - 评估样本
     - 模块归一化规则
     - 联动关系规则

---

## 6. 维护规则

- 新增一级模块，必须先更新本文件与配置文件
- 新增二级能力，必须明确挂在哪个一级模块下
- 新增模块别名，必须避免与已有模块冲突
- 调整模块边界时，必须同步更新：
  - 配置文件
  - 评估样本
  - 模块归一化规则
  - 联动关系规则
- 不允许只改代码、不改规则文档
- 不允许只改样本、不改模块定义
